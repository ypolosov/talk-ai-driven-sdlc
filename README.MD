*// Вводим: методы и роли*

9. **Концепция · Методы, Практики и Роли**
   - Метод / практика — способ выполнения работы
     (алгоритм, культура, стиль)
   - Роль — функциональная позиция агента,
     определяемая методом который он исполняет
   - Агент — исполнитель роли: человек или AI —
     не имеет значения, важен метод
   - Мастерство — способность агента исполнить метод
   - Главный принцип: разговаривай с ролями,
     а не с людьми
   - В software: «Архитектор» — это роль, не должность.
     Роль определяется методом (ADD 3.0, arc42),
     а не записью в контракте

10. **Концепция · Бизнес-модель (Business Model Canvas)**
    - Бизнес-модель описывает как организация создаёт,

11. **Концепция · Поток создания ценности (Value Stream)**
    - Value Stream — последовательность шагов,
      которая доставляет ценность от идеи до потребителя
    - Два вида (SAFe / Lean):
      - Operational Value Stream (OVS) —
        доставляет ценность конечному потребителю
        (пирожок → прохожему, игра → игроку)
      - Development Value Stream (DVS) —
        создаёт системы которые обеспечивают OVS
        (создание духовки, создание платформы)
    - Ключевой вопрос: где в цепочке создаётся ценность
      и для кого?

*// Bakery Factory через BM + Value Streams*

14. **Bakery Factory · Бизнес-модель 1 (B2B)**

    **Кто:** компания пекарного оборудования
    **Что продаёт:** духовки пекарям
    **Value Proposition:** промышленная духовка
    как инструмент производства пирожков

    **Development Value Stream (DVS):**
    проектирование → производство → продажа духовки

    **Operational Value Stream (OVS):**
    доставка → установка → обучение → поддержка пекаря

    **Системный разбор:**
    - ЦС: духовка
    - СС: фабрика + конвеер
    - Надсистема: пекарня пекаря
    - ЦС(духовка) = инструмент СС пекаря

15. **Bakery Factory · Бизнес-модель 2 (B2C)**

    **Кто:** пекарь
    **Что продаёт:** пирожки прохожим
    **Value Proposition:** свежая выпечка здесь и сейчас

    **Development Value Stream (DVS):**
    рецепт → закупка → подготовка

    **Operational Value Stream (OVS):**
    замес → выпечка → продажа → потребление

    **Системный разбор:**
    - ЦС: пирожок
    - СС: пекарь + духовка (инструмент)
    - Надсистема: прохожий

16. **Bakery Factory · Лабораторная модель**

    **Проблема BM1:** обратная связь о качестве духовки
    приходит через жалобы клиентов, задержка — месяцы

    **Лабораторная модель:**
    компания строит фабрику-лабораторию
    и лабораторию-пекарню

    **Добавляет собственный OVS:**
    выпечка пирожков → продажа прохожим

    **Value Stream петля:**
    духовка → лаборатория-пекарня → пирожки
    → прохожие → feedback → лучшая духовка

    **Ценность:** DVS (производство духовок)
    улучшается через обратную связь
    из собственного реального OVS

    **Operational Value Stream (OVS):**

30. **Концепция · Классический SDLC**
    - Software Development Lifecycle —
      жизненный цикл разработки ПО
    - Фазы: Vision → Requirements → Architecture →
      Development → Testing → Deployment → Ops
    - Каждая фаза — набор методов работы и артефактов
    - Агенты (люди) выполняют роли, используя инструменты

31. **Где классический SDLC теряет качество**
    - Решения умирают в Slack и в головах
    - Метод работы неявный → результат невоспроизводим
    - Связи между фазами рвутся:
      требование не трейсируется до теста,
      ADR не трейсируется до кода
    - Переход: нужен явный метод — это первые принципы

32. **Первые принципы (P-A..E)**
    - P-A: система создания определяет качество ЦС
    - P-B: метод работы должен быть явным и трейсируемым
    - P-C: сложность метода соразмерна сложности системы
    - P-D: эволюция через умные мутации описания,
      не хотфиксы
    - P-E: агент — исполнитель роли,
      роль определяется методом

35. **Концепция · Мемом и Феном (Левенчук)**
    - Мемом — описание системы: геном, живёт у создателей

      vs испечённый пирожок (феном)
    - В software: git-репозиторий (мемом)
      vs production (феном)
    - Изменять феном «напильником» дорого;
      изменять мемом — дёшево и безопасно

38. **Концепция · Continuous Everything · три времени**
    - Классический SDLC: два времени
      - Dev-time: мемом воплощается в феном
      - Ops-time: феном работает в production
    - Проблема: кто отвечает за эволюцию мемома?
      В классическом SDLC — никто явно
    - Continuous Everything добавляет третье время:
      - Evo-time: мемом намеренно эволюционирует
        через умные мутации на основе наблюдений
        из Ops-time
    - Именно Evo-time предотвращает architectural drift

41. **Canonical Evolution Loop**
    - Ops-time: феном работает, мы наблюдаем
      (метрики, инциденты, user feedback)
    - Сигнал: нарушение fitness condition
      или появление новой возможности
    - Evo-time: DRR — аргументированная умная мутация
      мемома (новый ADR, обновлённый SKILL.md)
    - Dev-time: обновлённый мемом воплощается
      в новый феном
    - Ops-time: наблюдаем снова → цикл замкнулся

    **Лабораторная модель в этом цикле:**
    - Тестовый сайт / лаборатория-пекарня =
      ускоритель Ops-time
    - Сигналы для Evo-time поступают
      до попадания к реальным клиентам

    **Хотфикс без DRR = мутация без аргументации
    = накопление долга мемома**

      это структурное требование
      к зрелой системе создания

90. **Три вещи которые можно сделать завтра**
    - Назвать свою ЦС по принципу почтальона
    - Написать один ADR в DRR-структуре
    - Создать один SKILL.md для самого частого
      метода в команде
